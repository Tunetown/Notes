Usage Documentation

This documentation is structured to give a short walkthrough, to make a first trial with a local database easier, before decumenting the features of the app. The chapters about local installation, making the app available offline and some other setup stuff are located at the end of this document, which is intentional because installation is optional and can be carried out at any time without any inconsistencies. To just see how it works in general, just read on.

Notebooks

The app stores all meta data and information for a notebook inside a CouchDB database, so we have to specify the address of the database first. When you load the landing page of the app, you normally see all notebooks opened in the past on this device. When you are new, only the "local" notebook is visible. 

This page is always available in the global user menu at the top right, too, in case you get lost:

NOTE: If you want to test the app using the local notebook, select it and skip the rest of this chapter and continue with "Create a Textual Note".

To create a new notebook on a remote CouchDB server, just create a new, empty database on it. Click "Add Notebook from CouchDB URL..." and provide the URL to the database:

An empty notebook with default settings will be created. You will be asked for credentials to access the database, if necessary. Provide them and you see an empty notebook like this:

Here, you see the navigation panel on the left, as well as the document area on the right. On mobile devices, only the navigation area is shown initially. 

To access a notebook from different devices, just do these very same steps on each device, just providing the URI of the database. All data from the notebook is then initially loaded from the server, so your notebook will look the same on all devices, and will automatically sync all changes with the server, as described in the later chapter about synchronization. Here, we will for now stay focused on the actual usage of the app first.

Authentication

This is handled by the CouchDB server exclusively. The app itself is just a static bunch of open source files, so there is no need for authentication in this place, everyone can open the app and work locally with it. However, the time you access a remote CouchDB database which needs authentication, you have to provide the credentials. The app will ask you for them when you are not logged in already, and it depends on the server how long this login is valid. The app uses the standard token mechanisms supported by PouchDB/CouchDB. 

NOTE: CouchDB uses its own mechanisms for authentication, other ways than Cookie Authentication are currently not supported! You have to create users on the database server admin site, and assign them to the database you need directly or via roles. See the CouchDB documentation on the topic for more details.

Text Search

The whole notebook can be searched by text search. Type in the search text above the navigation bar, and the navigation will show the results immediately after each key you type. This is very fast because the whole notebook data is held locally, no network bandwidth is needed for this. 

The search includes the item name, content and labels, basically everything. It does NOT search inside attachments like PDFs however.

Document History

Every document (exceptAttachments) has a document history. This means that every time a change is saved, a new version is created. To avoid too much versions, they are also deleted again after a while, which is done at the time of saving: If you save a document a few days later for example, some of the version are deleted which have been created the time before. The algorithm works like this at the time of saving a document:

  • Do not delete anything newer than 1 minute
  • Keep one version for the first 10 minutes
  • Keep one version each 10 minutes for the first hour
  • Keep one version each hour for the first day
  • Keep one each day for older notes

NOTE: YOu can switch off the deletion of versions in the Settings page. This is not recommended because it produces much redundant data however.

The history is shown along with the change protocol, which is also written at each change of the document:

You can delete single entries as well as all at once, and of course see the content of the versions. Click on the eye icon to see the version data, and copy/paste the content back to your document if you need it.

NOTE: The "revoke" action is still buggy and will be re-enabled in the future to provide direct revoking of old versions without copy/paste.

NOTE: See method ui/app/src/data/Document.reduceVersions() for technical details.

Settings Page

This is accessible through the global user menu and shows the notebook settings. The following chapters of options are available:

Remote Connection Settings

This shows options and information about the opened notebook. Also you can manage the notebooks available on the device/clients landing page. At the top, you can choose the notebook opened among the ones which are available, and you can add new notebooks by URL here, too, just as in the landing page.

"Close Notebook..." will remove currently opened the notebook from the client memory. It will not be available for selection in the landing page anymore. The remote data is not changed, only the local mirror on the device is cleared.

NOTE: These settings are stored on the client side and are individual for all client devices.

Remote Sync Settings

This is described in the later chapter about Synchronisation Setup.

NOTE: These settings are stored on the client side and are individual for all client devices.

Import/Export of Documents

These are not really settings, but provide options to get data in and out of the app.

Here you have the options available globally for the notebook. You can set the book name, colors, default editors/languages, the interval for auto saving, and some other options. Some of them can be set individually for desktops and mobile devices.

NOTE: These settings are stored on the client side and are individual for all client devices.

Synchronisation Setup

Making the Notebook Available Offline

If you set up a notebook which is connected to a remote database, at first, the database is only available when there is a network connection to it. All changes are communicated directly to the server, there is no local buffer.

To make the app available when offline, we have to go into the settings page, and check the chapter about "Remote Sync":

Per default, the two switches are set to Off. 

  • Enable the "Make available offline" option, which will create a local database first. However, this database will be empty now (no panic, your documents are still there!), because the app is not syncing automatically:
  • Enable "Auto sync to server" will activate the automatic alignment after each change, may it be on the server or in the app. After activating this, you will see your documents again, as the app did a sync now.

NOTE: You can also leave the second option off, but you have to manually trigger synchronization via the Header Bar then, which is not very convenient.

Other options are:

Check Consistency

This opens a dialog which checks in depth if the local and remote notebook instances are in sync and consistent, mainly for debugging purposes in Development:

Run all check with the button on the top right. YOu will get detailed messages if anything is not OK, and for much cases there is also an automatic fix, which can be triggered by these buttons:

In this case, we have a problem with the internal views, which is totally uncritical and relates to an old version of the app previously used. Clicking the Cure button will delete them automatically. You could also ignore this, it is just a Warning and has no influence on the usage of the notebook.

Replicate Local to Remote

This replicates the local database mirror to any remote server you like. You can clone your notebook completely to a new server using this for example. You have to specify the URI of the database to be filled.

Clear Local Data

This initiates the local mirror database completely. It will be replicated again from the remote if automatic sync is active.

Conflicts

If you are offline (for example) and change a document, while on another client, the same document is also changed, you have a conflict: The app cannot determine which version is the valid one. This conflict solving is supported by CouchDB itself, see the documentation for deeper technical details.

However, for the user, whenever a conflict is evident after syncing, an error symbol will indicate this for the document in question:

In the global user menu, you can see all conflicts:

If you click on the document name (here: "Example", you see the document which the CouchDB algorithm has chosen to be the main one. This is the content you see when opening the document, just ignoring the conflicts:

If you click on the Eye icon in the line, you see the content of the concurrent document version, along with a detailed analysis of which parts have changed (diff):

Here you see that in this example, the offline client wrote "Content23" while the online version removes the "23" and adds the "4" to the new content "Content4". This makes analysis easier.

Resolution of Conflicts

If the databse chose the correct version already, just delete the conflict in the list. 

If the database chose the wrong version to be correct, open the conflict version which is right as described above, by using the eye icon button. In the following page, you have the option to set this version as the correct one ("Set as Winner"):

After this, the conflict is gone and the document shows the right content, in this case "Content4". You have to save this before it will be valid! Also, the conflict still exists as we just overwrote the content, so go back into the conflict list and delete it.

Installation as PWA

The app uses mechanisms of the Progressive Web App specification to provide full functionality also when no internet connection to the server can be established. This does not need any installation and is working from the start, also in the browser (if you set up the notebook for being available offline of course, see above chapters. This chapter just means the application itself).

However, if you want to have the feel of a real app installed on the device, you can do this with the PWA specification. See this artcle for example or just search for how to install a progressive web app, this is just one or two clicks on most platforms. It then can run in a separate window from the browser (desktop) or like a genuine mobile app, making the usage much more convenient. 

You can decide to install at every time, the data will be taken over automatically, even with local notebooks.
