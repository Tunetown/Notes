Overview

The code base has grown over time, but still is pretty structured, however some classes need to be split again. One script file always just contains one class, no exceptions. The app itself is completely static, consisting of HTML, CSS and JS files, served by anything that can serve simple web pages. It connects to the CouchDB server directly from the client side.

Internally, index page only contains some modal forms as well as the script/stylesheet references to all needed stuff, but the app itself is completely built by JS using JQuery. This is the main entry point in index.html:

The sources are located in the ui folder:

  • Notes is the main application controller class.
  • Config is a static class defining some options.
  • Routing is responsible for the app internal routing, built upon sammy.js.

Folders described:

  • lib: External JS libraries
  • tools: Internal helpers, for example to create the app icons. This is not part of the app itself.
  • app: The main sources of the app
    • actions: contains all routines which perform changes etc. All actions are implemented to return Promise objects.
    • data: Classes for data handling (model classes etc.)
    • database: Wrapper classes for handling the PouchDB library to achieve synchronisation, authentication etc.
    • import/export: Importer and Exporter Classes
    • navigation: This holds the implementation of the navigation area at the left.
    • pages: One class for each page available (note editors, trash bin, version history, settings, conflicts, ....)
    • tools: Miscellaneous tool classes

Notes, Actions and Routing as well as some other classes use a singleton approach, so for example access to the main application instance can anywhere be achieved using this syntax:

The whole app works with Promises to achieve the asynchronous tasks needed. There is no minimized version or anything. Lots of comments are in the source code also, because maintainability is considered more important than performance optimization for all costs. Because of the offline capable nature of the app, the sources will also be loaded just once and then served from the service worker script, so this does not play a major role here anyway.

Icons are provided by Font Awesome.

Progressive Web App Details

The PWA specification defines a Service Worker script, which is installed at the first loading of the app, and then serves the app's scripts locally if in cache. Also there is a PWA manifest file, responsible for the display options when installed as a PWA (icons, name etc.).

The service worker must contain entries for all files needed by the app (not only those also linked in the index file, but also those loaded dynamically by scripts etc.). If you miss to include a needed file in the SW, the app will have errors when offline! If you need to include new files for offline usage, add them here:

Also, the name of the so called Pre Cache shall be changed whenever there has been an update to the app, to prevent deprecated sources to be loaded.

The registration and auto aupdate mechanisms for the SW is located in the Notes class, method registerServiceWorker(). 3 Seconds after launching the app, it automatically checks if a new SW is available and schedules it for installation. However, sometimes you need to manually stop the old service worker to get the new one installed properly (you can do this in Chrome for example using the developer tools (F12), Tab "Application" -> "Service Workers" and stop the active one there:

Adding new Scripts

To add new files to the app, may it be CSS or JS or whatever is also necessary offline, the following has to be done:

  • Add the file to index.html as script or stylesheet or load it dynamically by JS
  • Add the file to the service worker pre cache (see above)
  • Change the SW pre chache name as described above (either update the serial number or, if it stays the same, increase the postfix)

Routing

The app uses a routing library called sammy.js. This library uses a completely URL based approach (much like REST), which parses the part of the URL after the hash. This way, every page can have a distinct URL you can directly use to get to the page again, even in a JS based web app. The Routing class defines all available routes of the app in its setup() method. 

Database Structure

CouchDB is a schemaless database, storing JSON documents of any structure just by their IDs. An example of a small database (the one created for this documentation and to generate the screen shots) looks like this in the CouchDB Admin page:

Besides the documents themselves (which use a random hash postfixed variant of the document name, which stays the same even if the visible name is changed), there are some technical documents:

  • settings: This contains the settings of the notebook (see Settings chapter in the Usage Documentation)
  • _design/views_X.XX.XXX: these are so called design documents, which CouchDB uses to define MapReduce views used to query the data. The views are maintained automatically, see the Database.createOrUpdateViews() method. The MapReduce views themselves are defined in Document.getViewDefinitions().

Document Structure

An example document looks as follows:

Fields:

  • _id: The "official" CouchDB ID of the document. Must be unique in the database (this is the reason for the random hash postfix).
  • _rev: In CouchDB, this denotes the revision of the document, which is used for conflict detection.
  • name: Display name of the note.
  • type: Type of Document. Value can be "note", "reference", "attachment" or "sheet" (see Document types in the Usage documentation)
  • order: The numerical order for placement in the level of hierarchy. This is updated when the position of the item is changed.
  • content: The content. In case of the code/plain text editor, this is the plain content. In case of the TinyMCE WYSIWYG editor, the content is stored as HTML.
  • timestamp: Last changed time stamp
  • parent: This defines the hierarchical structure of the notebook, and defines under which item this item should be shown, by its _id. Note that if the parent reference is not found anymore, th item becomes unaccessible in the navigation! This will be detected by the consistency check however, see Usage documentation.
  • changeLog: This is a list containing the change log for the document as JSON array. This is what is shown in the history page.
  • editor: The editor mode. Value can be "richtext", "code", or "board"
  • links: This stores all links contained in the document's content (just for type 'note'). This is updated every time a document is saved, and mainly serves the purpose of speeding up the data algorithms.
  • attachmentSize: Size of all attachments. This just speeds up the info determination and is updated on every saving of the document.
  • contentSize: Size of the content data. This just speeds up the info determination and is updated on every saving of the document.
  • changeLogSize: Size of the whole change log. This just speeds up the info determination and is updated on every saving of the document.
  • preview: Generated preview, derived from the content, and shown for exampe in the Detail List navigation mode.
  • _attachments: This is a CouchDB internal field, and contains attachments to the document. This can be:
    • Attachment data in reference documents. The actual data is omitted in the above view, this is done by CouchDB automatically.
    • Versions of the document, which are also stored as attachments of the type "text/html" prefixed with "version_" internally.

You see, all data regarding one document is in one place. This makes the database structure overseeable.

Navigation

The navigation is built upon the Muuri framework (as well as the Kanban editor page, too).

The common stuff is implemented in NoteTree.js, which is kind of the controller of the navigation panel. It uses the MuuriGrid class, which is a wrapper around the Muuri library, to generate the grids. The other classes are:

  • Behaviours: Manages the switching of navigation modes (here called "Behaviours"), and also creates the behaviour instances (Factory).
  • ExpandedState: Implements expansion and collapsing state persistence
  • Scrollstate: Manages the scroll state persistence

Behaviours are defined as separate classes as well, which all must have exavlty the same public methods (which are many, so please refer to the code comments and usage of the methods in NoteTree.js). You can define your own, new behavior by copying one of them, modifying it and add it in the functions contained in Behaviours.js.
